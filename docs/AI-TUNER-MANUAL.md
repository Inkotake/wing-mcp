## 1. 什么是 AI 调音师

AI 调音师是一个基于大语言模型的智能音频助手，专门为 Behringer WING / WING Compact / WING Rack 数字调音台设计。它放在你的排练室里，连接到 WING 调音台，你可以通过语音或文字跟它对话。

**它不是简单的"命令翻译器"，也不是自动混音器。** 它是一个能像专业音响工程师一样思考的智能系统——懂调音台、懂排查逻辑、但必须经过你确认才会动手。

### 核心特点

- **诊断优先，修改在后**：AI 会先读取调音台状态、看电平表、查路由，找到问题断点后再提修复建议
- **安全第一**：所有写入操作都经过严格的安全策略审核，高风险操作必须由你亲自确认
- **只做你有把握的事**：一次只让你做一个动作，不会甩给你十条清单
- **记住你的场地**：能记住房间设备连接、乐队偏好、历史故障
- **支持语音对话**：你可以像和对讲机说话一样和它交流（需配置语音模块）

### 工作方式

```
你说："主唱没声音"
    ↓
AI 听 → 理解意图 → 读取调音台状态 → 分析问题 → 给你建议
    ↓
你确认 → AI 执行修改 → 回读验证 → 记录审计日志
```

系统架构：

```
你的声音 / 键盘输入
     ↓
语音助手 / 聊天界面
     ↓
AI 引擎（模型路由 + 诊断引擎 + 房间记忆）
     ↓
MCP 服务器（WING 控制 + 安全策略 + 审计日志）
     ↓
Behringer WING 调音台
```

## 2. 能做什么

### 2.1 没声音诊断（No-Sound Diagnosis）

你说某个通道/监听/主扩没声音，AI 会系统性地排查：

1. **信号源**：话筒/乐器是否在工作？无线接收器有信号吗？
2. **输入路由**：XLR 线插对了吗？patch 到正确的通道了吗？
3. **通道状态**：通道是否静音？推子是否拉下？噪声门是否关死？
4. **发送/母线**：到监听母线的发送量够吗？母线是否静音？
5. **主输出**：Main LR 静音了吗？推子正常吗？
6. **输出路由**：输出 patch 对吗？功放/音箱开了吗？

每一步都用实际的 meter 读数和路由追踪来验证，不是瞎猜。

### 2.2 监听混音调整（Monitor Mix）

- 帮乐手调整耳返/地板箱里的监听比例
- "鼓手耳返里加一点主唱" → 定位到正确的通道和母线，准备调整发送量
- "耳返里吉他太大了" → 减少对应的 send level
- 注意：耳返音量过大可能损伤听力，AI 会限制单次调整幅度

### 2.3 反馈/啸叫处理（Feedback Handling）

- 检查监听音量是否过高
- 检查话筒和音箱的相对位置
- 检查 EQ 是否有共振频率
- 建议但不盲目修改 -- 啸叫处理依赖于现场判断

### 2.4 路由帮助（Routing Help）

- 追踪信号从源头到输出的完整路径
- 检查 input patch、output patch 是否正确
- 路由修改是 critical 风险，需要你明确确认

### 2.5 演出前 Line Check

- 逐通道快速检查：是否有信号？是否静音？推子位置？
- 主输出状态检查
- 生成检查报告

### 2.6 场景/快照管理

- 查看当前场景和可用场景列表
- 保存当前状态为快照
- Recall 场景（高风险，需明确确认）

### 2.7 效果器管理

- 查看 8 个 FX 槽位当前使用的效果器型号
- 切换效果器类型
- 调整效果器参数

### 2.8 EQ 和动态处理

- 查看和调整通道 EQ（4 段参数均衡器）
- 查看和调整噪声门（Gate）
- 查看和调整压缩器（Compressor）

### 2.9 房间记忆

- 记住每个房间的设备连接方式
- 记住每个乐队的通道分配
- 记住历史故障和解决方法
- 记住乐器清单和话筒偏好

## 3. 现场安全规则

### 核心原则

> **安全在系统，不在提示词。**

AI 调音师的安全策略写在 MCP 服务器代码里，不是依赖模型"自觉"。无论你怎么说话、怎么请求，危险的操作都必须经过验证。

AI 调音师有几个不可绕过的安全规则：

1. **诊断先于修改** — 总是先查状态再提建议
2. **一次只做一个动作** — 不会一口气让你做十件事
3. **高危操作必须精确确认** — 幻象电源、路由、场景切换需要你说完整的确认语句
4. **检测到回读不匹配自动停止** — 如果写入后的读值与目标不一致，立即报告
5. **演出模式默认只读** — 演出中可以查看状态但不能修改
6. **Raw 协议默认禁用** — 不能绕过安全层直接发送原始命令

### 风险等级

| 等级 | 含义 | 示例 | 需要确认？ |
|------|------|------|-----------|
| **none** | 纯读取，无影响 | 查看状态、读 meter、搜索参数 | 不需要 |
| **low** | 装饰性修改 | 改通道名称、颜色 | 不需要 |
| **medium** | 影响单个通道 | 调整通道推子、静音/取消静音、调整发送量、调整 EQ | 需要确认 |
| **high** | 影响多个通道或主输出 | 调主输出推子、DCA 静音、mute group、噪声门、FX 切换 | 需要精确确认 |
| **critical** | 可能损坏设备或导致全场无声 | 幻象电源、路由修改、场景 recall、raw 协议命令 | 需要精确确认 + 风险知情 |

### 写入操作的完整流程

每个修改都经过 13 个步骤：

```
1. 解析目标参数（把"主唱"翻译成 CH 1）
2. 读取当前值（确认要改什么）
3. 评估风险等级（RiskEngine）
4. 检查安全策略（PolicyEngine）
5. 生成修改计划（ChangePlanner）
6. 返回确认请求（如果需要）
7. 等待你的精确确认
8. 再次读取当前值（确认没被别人改过）
9. 执行修改
10. 回读验证
11. 对比目标值和实际值
12. 写入审计日志
13. 用人话告诉你结果
```

### 修改幅度限制

| 操作 | 排练/演出模式下最大幅度 |
|------|----------------------|
| 通道推子调整 | 一次最多 3 dB |
| 发送量调整 | 一次最多 6 dB |
| 主输出推子 | 一次最多 1.5 dB |
| EQ 增益调整 | 一次最多 3 dB |
| 噪声门阈值 | 一次最多 6 dB |
| 每分钟写入次数 | 最多 12 次 |

### 绝对禁止的操作

以下操作无论你怎么说都不会执行：

- 演出模式（live mode）下的 raw OSC/native 协议命令
- 未经精确确认的 critical 操作
- 使用过期的确认 ID
- 用 A 操作的确认 ID 去执行 B 操作
- prepare 之后被其他人修改过的参数（会要求重新 prepare）
- 有未完成的诊断会话时 recall scene（除非管理员覆盖）

## 4. 运行模式说明

AI 调音师有四种运行模式，由管理员设置：

| 模式 | 能做什么 | 什么时候用 |
|------|---------|-----------|
| **只读模式** (read_only) | 只能查看，不能修改任何参数 | 演出进行中、第一次接入、访客观摩 |
| **排练安全模式** (rehearsal_safe) | 可以调整通道推子、静音、发送量、EQ（需确认）；高危操作被拒绝 | 排练、试音、调试 |
| **维护模式** (maintenance) | 可以做所有操作，包括路由、幻象电源、场景切换（需精确确认） | 空场、不上 PA、检修 |
| **开发者模式** (developer_raw) | 包括 raw OSC/native 协议命令 | 仅在开发调试时使用，禁止演出时启用 |

**你在使用时不需要手动切换模式**。系统会根据当前模式自动允许或拒绝你的请求。如果请求被拒绝，AI 会告诉你原因。

## 5. 实战场景

### 场景一：主唱没声音

**你说：** "主唱没声音"

**AI 调音师会怎么做：**

```
第一步：明确范围
AI：好的，让我先检查一下，不会改调音台。
    [读取 Room A patch sheet] 主唱 → CH 1 (Vocal 1)
    主唱是 CH 1 对吗？

第二步：读取状态
AI 自动调用工具：
  - wing_channel_get(1) → 读取 CH 1 当前状态
  - wing_meter_read(["/ch/1/fader"]) → 看有没有输入电平
  - wing_signal_path_trace(1) → 追踪信号从输入到输出的完整路径

第三步：根据发现给你反馈
```

**情况 A：输入 meter 无信号**

```
AI：CH 1 没有输入信号 (RMS -65 dBFS)。
    通道路径正常：Headamp gain 30dB, 未静音, 推子 -6dB。
    问题可能在音源端。

    请检查：
    1. 主唱的话筒线是否插好
    2. 无线接收器是否开机、有没有信号灯
    3. 话筒开关是否打开
    不用动调音台。

（你检查发现话筒线松了，插紧后）

你：线松了，现在插好了。

AI：[重新检查] CH 1: RMS -18.3 dBFS, 有信号！置信度 92%。
    信号恢复了！当前推子在 -6.0 dB。还需要调整什么吗？
```

**情况 B：输入 meter 有信号，但通道被静音**

```
AI：CH 1 有输入信号 (RMS -22 dBFS)，但通道被静音。
    准备操作：CH 1 静音从 ON 改为 OFF
    风险等级：medium
    请确认："确认解除 Vocal 1 静音"

你：确认解除 Vocal 1 静音

AI：✅ 已完成：CH 1 mute 从 ON 改为 OFF；WING 回读 OFF。
    审计编号 aud_123。现在应该有声音了。
```

**情况 C：通道正常，但 Main LR 没信号**

```
AI：CH 1 信号正常，通道未静音，推子在 -6 dB。
    但 Main LR 没有输出信号。
    需要我检查 Main LR 状态吗？

你：检查

AI：Main LR 被静音了！
    准备操作：Main LR 静音从 ON 改为 OFF
    风险等级：high（会影响整个 PA）
    请确认："确认取消 Main LR 静音"
```

**情况 D：Main LR 有信号，但房间没声音**

```
AI：调音台内部信号通路正常，从话筒到 Main LR 都有信号。
    Main LR meter 显示 RMS -18 dBFS。
    问题可能在外部设备：功放有没有开？音箱电源？处理器静音？
    请检查功放和音箱，这超出了调音台的范围。
```

**关键原则：从不瞎改调音台。先读后想再说。**

### 场景二：鼓手耳返里听不到吉他

**你说：** "鼓手耳返里听不到吉他"

**AI 调音师会怎么做：**

```
第一步：确认目标
AI：让我检查当前状态。我先确认一下目标——
    [查找] 吉他 → CH 3, 鼓手耳返 → Bus 1 (Drummer IEM)

第二步：读取状态
AI 自动调用：
  - wing_send_get(3, 1) → 读取 CH 3 到 Bus 1 的发送量
  - wing_channel_get(3) → 确认 CH 3 是否有输入信号
  - wing_meter_read(["/bus/1/fader"]) → 看 Bus 1 有没有信号

第三步：根据发现反馈
```

**情况 A：发送量为 -99 dB（没发送）**

```
AI：[读取发送量] CH 3 → Bus 1: -99.0 dB ← 等于没送！

    诊断：CH 3 有输入信号，但到 Bus 1 的发送量为 -99 dB，
    鼓手几乎听不到。

    建议：将 CH 3 → Bus 1 发送量从 -99 dB 提高到 -10 dB。
    风险等级：medium（监听发送调整）
    ⚠️ 提醒：这会改变鼓手的耳返混音比例

    请确认："确认把吉他调到鼓手耳返 -10 dB"

你：确认把吉他调到鼓手耳返 -10 dB

AI：✅ 已完成：CH 3 → Bus 1 从 -99.0 dB 调到 -10.0 dB；
    WING 回读 -10.0 dB。审计编号 aud_456。
    鼓手现在应该能听到吉他了。需要进一步调整吗？
```

**情况 B：发送量正常，但 Bus 1 被静音**

```
AI：CH 3 到 Bus 1 的发送量是 -8 dB，正常。
    但 Bus 1（鼓手耳返）本身被静音了。

    准备操作：Bus 1 静音从 ON 改为 OFF
    风险等级：medium
    请确认："确认取消 Bus 1 鼓手耳返静音"
```

**情况 C：一切正常，但鼓手说听不到**

```
AI：CH 3 有信号，发送量 -8 dB 正常，Bus 1 未静音。
    调音台这边看起来没问题。

    可能是鼓手的耳返接收器音量不够，或者耳返没戴好。
    请让鼓手检查：耳返接收器是否开机？音量旋钮是否太小？
```

**关键原则：耳返音量调整需谨慎，一次幅度不超过 6 dB。**

### 场景三：有啸叫

**你说：** "有啸叫"

**AI 调音师会怎么做：**

```
第一步：了解情况
AI：让我检查一下。先不修改任何设置。
    哪个位置有啸叫？主扩还是监听？

你：主唱话筒对着主扩的时候会叫

第二步：读取状态
AI 自动调用：
  - wing_state_summary → 获取整体状态
  - wing_eq_get("ch/1") → 查看 CH 1 EQ
  - wing_send_get(1, 2) → 检查发送到 Vocal IEM 的量

第三步：给出建议

AI：我发现了以下情况：

    1. CH 1 发送到 Bus 2 (Vox IEM) 的发送量是 -5.0 dB — 比较高
    2. CH 1 EQ high gain: +4.0 dB — 高频提升可能加重了反馈
    3. 话筒（心形指向）可能正对着主扩音箱

    建议（按优先级，从最安全到最危险）：
    1. [不用动调音台] 让人把话筒和音箱的相对位置调整一下
       不要让话筒正对音箱

    2. [medium 风险] 如果还是叫，可以把 CH 1 high EQ gain
       从 +4 dB 降到 0 dB，减少高频共振

    3. [medium 风险] 最后的办法是降低 Bus 2 (Vox IEM) 的发送量

    先从第一步开始？不需要我修改任何设置。
```

**关键原则：啸叫处理依赖现场判断，AI 不会盲目修改 EQ。**

## 6. 你该怎么说

### 和 AI 沟通的技巧

**说清楚问题，而不是直接说操作。**

| ✅ 这样说 | ❌ 不要这样说 |
|----------|-------------|
| "主唱没声音" | "把 CH 1 mute 关掉" |
| "鼓手说耳返里吉他太小了" | "CH 3 send 1 加 6dB" |
| "有啸叫，好像是主扩" | "拉低 Main LR" |
| "帮我做 line check" | "把每个 mute 都关了" |
| "直播没有声音" | "改 Matrix 1 路由" |

**为什么？**

AI 调音师需要先诊断问题再修改。你直接说操作，AI 会跳过诊断步骤，可能错过真正的问题原因。比如 CH 1 没声音可能是因为 XLR 线松了，而不是通道静音。

### 基本对话模式

```
你   → 描述问题
AI   → 确认范围（"主唱是 CH 1 对吗？"）
AI   → 读取状态，分析问题
AI   → 告诉你发现了什么
AI   → 如果有修复方案，告诉你准备做什么，风险是什么
你   → 确认（或否定）
AI   → 执行修改，告诉你结果和审计编号
```

### 确认用语

AI 需要确认时，会根据风险等级要求不同精度的确认：

**Medium 风险**（通道推子、静音、发送、EQ）：
```
确认执行
确认把主唱推高 3dB
确认取消 CH 1 静音
```

**High 风险**（主输出、DCA、mute group、噪声门、FX 切换）：
```
确认把 Main LR 降低 1dB
确认 mute DCA 2
```

**Critical 风险**（幻象电源、路由、场景 recall）：
```
确认开启 LCL.1 的 48V 幻象电源，我确认连接设备需要幻象电源
确认 recall Scene 12，我知道这会改变当前调音台状态
确认修改 Room A 的输出路由，我知道可能导致主扩或耳返无声
```

### 理解 AI 的反馈

AI 反馈里可能包含以下信息：

```
✅ 已完成 — 操作成功
⚠️ 警告 — 有需要注意的问题
❌ 拒绝 — 操作被安全策略阻止
🔐 确认ID — 写入操作需要确认
📋 步骤 — 诊断中的当前步骤
🔍 检查中 — AI 正在读取状态
```

## 8. 常见问题（FAQ）

### Q1：AI 会不会乱动我的调音台？

**不会。** 所有修改都需要你的明确确认。高危操作（幻象电源、路由、场景切换）需要你说完整的确认语句，系统才会执行。AI 会先告诉你它发现了什么、准备做什么、风险是什么，然后等你确认。

### Q2：排练时应该用什么模式？

**rehearsal_safe。** 允许你调整推子、静音、发送量、EQ，但禁止高危操作（路由、幻象电源、场景切换）。这是最常用的工作模式。

### Q3：演出时能用吗？

**可以，但建议设为 read_only 模式。** 演出进行中，AI 可以看到所有状态、回答"有没有问题"、帮你做诊断，但完全不能修改任何参数。这是最安全的用法。

### Q4：需要联网吗？

**MCP 服务器在你本地运行。** AI 模型（Claude/ChatGPT）需要 API 连接来理解你的请求。如果是 fake-wing 模式（虚拟调音台），可以完全离线使用来做测试和学习。

### Q5：没买 WING 能用吗？

**可以！** 用 `WING_DRIVER=fake` 模式启动 AI，它会加载一个虚拟的 48 通道 WING 调音台。你可以体验所有功能，包括诊断、调整、line check 等，不需要任何硬件。

### Q6：支持哪些 WING 型号？

全部支持：
- WING（标准版）
- WING Compact
- WING Rack

三个型号使用相同的控制协议，AI 会通过 discovery 自动识别型号。

### Q7：AI 能不能帮我混音？

**不能，也不应该。** AI 调音师是诊断和辅助工具，不是自动混音器。混音是艺术创作，需要你的耳朵和判断。AI 可以帮你解决技术问题（没声音、路由错误、啸叫）和按你的要求做调整（"吉他大一点"），但不会替你做混音决策。

### Q8：AI 会记住我的场地设置吗？

**会。** 如果你配置了 sound-memory-mcp，AI 能记住：
- 房间设备的连接方式（哪个输入口接了什么）
- 乐队的通道分配（主唱是 CH 1，吉他 CH 3...）
- 历史故障和解决方法
- 乐队偏好（鼓手耳返里需要多少 click...）

这些记忆需要用专门的工具写入，AI 不会自动猜测并写入记忆。

### Q9：AI 说"操作被拒绝"是什么意思？

你的请求违反了安全策略。常见原因：
- 当前是只读模式，不允许写入
- 请求的修改幅度超过限制（比如一次想推高 10 dB）
- 高风险操作在排练模式下被禁止
- 确认 ID 过期或不匹配

如果你确实需要执行这个操作，联系管理员切换到维护模式。

### Q10：AI 说"回读不匹配"是什么意思？

AI 写了一个值到调音台，但读回来的值和预期不一致。这是一个安全机制——如果值没写成功，AI 不会假装成功了，而是告诉你出现了问题。重新 prepare 一次通常可以解决。

### Q11：可以用语音控制吗？

**可以，但需要额外配置。** voice-shell 模块支持 push-to-talk 语音对话。第一版推荐 push-to-talk（按键说话），避免在嘈杂环境中误触发。语音输出会路由到工程师监听耳机或小音箱，不会进入主扩或乐手监听。

### Q12：支持哪些语言？

AI 可以理解中文和英文。你可以用中文说"主唱没声音"，也可以用英文说"main vocal has no sound"。参数搜索也同时支持中英文关键词。

## 9. 紧急情况处理

### 如果 AI 做了你不期望的操作

**物理切断是最可靠的方法：**

1. **调音台端**：操作 WING 实体推子恢复到期望状态 — 实体操作始终优先于远程控制
2. **网络端**：拔掉 AI 设备与调音台之间的网线，断开控制连接
3. **软件端**：`killall node` 或 `systemctl stop wing-console-mcp` 停止 MCP 服务

### 如果怀疑 AI 出了问题

1. 立即让管理员切换到 `read_only` 模式
2. 检查审计日志，查看所有操作记录
3. 从 WING 直接 recall 之前保存的场景备份

### 预防措施（每次演出前必做）

```
[ ] 保存当前场景备份到 WING 内部存储
[ ] 将 AI 设为 read_only 模式
[ ] 确认 TTS 输出不路由到 PA
[ ] 确认 raw 工具已禁用
[ ] 确认网络连接稳定（有线优先，不用 Wi-Fi）
[ ] 确认有人知道如何物理断开 AI 连接
[ ] 确认 WING 实体推子的操作权限优先于 AI
[ ] 确认输出音量在物理上安全可控
```
